# AGENTS.md

## プロジェクト概要
このリポジトリは、対面プレイ用の『タイムボム』WebアプリのMVPを実装するためのものです。
プレイヤーは各自のスマホを自分の前に置き、秘密情報は自分の端末だけで確認します。

詳細仕様は `docs/timebomb-mvp-spec.md` を最優先で参照してください。
このファイルに書かれていることと実装詳細が食い違う場合は、仕様書を優先してください。

## 実装の目的
まずはMVPとして、4〜8人で最後まで遊べる状態を作ってください。
UIの豪華さより、ゲーム状態の正しさ・同期の正しさ・秘密情報の分離を優先してください。

## 優先順位
1. ゲームロジックの正しさ
2. サーバー主導の状態管理
3. 秘密情報が他プレイヤーに漏れないこと
4. リアルタイム同期の安定性
5. UIの見やすさ
6. 演出や見た目

## 必須要件
- 4〜8人でプレイできること
- 参加コードで入室できること
- 名前重複を禁止すること
- ゲーム開始後は参加不可にすること
- スパイなし
- オオタさんなし
- 人数ごとの役職・導線構成は仕様書どおりに実装すること
- 役職再確認と導線再確認を実装すること
- カード切断時に確認ダイアログを出すこと
- ラウンド進行と勝敗判定をサーバーで管理すること
- 同端末リロード復帰を実装すること

## 禁止事項
- ローカルルールを勝手に追加しない
- 仕様にない設定項目を増やさない
- 他プレイヤーの未公開情報をクライアントに送らない
- 勝敗判定をクライアント側に持たせない
- 2人・3人プレイを追加しない
- 中央共有画面を追加しない
- 永続化や戦績管理を先に入れない

## 技術方針
以下は推奨。既存リポジトリの事情があればそれに従ってよい。
- TypeScriptを使う
- フロントエンドとサーバーで型をなるべく共有する
- リアルタイム同期はWebSocketベースで行う
- 真実のゲーム状態はサーバーのみが持つ
- クライアントは表示と入力に集中させる

## 実装スタイル
- 小さく動く単位で進める
- 先にデータ構造と状態遷移を安定させる
- UIを作る前に、ルーム管理・ゲーム状態管理・勝敗判定を実装する
- 1つの作業を始める前に、何を変更するか短く計画を書く
- 大きな変更のあとには、必要なら関連する型・テスト・ドキュメントも更新する

## テスト方針
最低限、以下のケースを確認してください。
- 4人でゲーム開始できる
- 8人でゲーム開始できる
- 役職未配布がある人数で正しく配布される
- BOOM公開で即終了する
- 解除全達成で即終了する
- 4ラウンド終了でボマー団勝利になる
- ホストだけが配り直しできる
- 開始後参加できない
- 名前重複で参加拒否される
- 同端末リロード復帰できる

## 期待する進め方
1. まず現状のコードベースを確認する
2. 実装計画を短くまとめる
3. MVPに必要な最小単位から順番に実装する
4. 各段階で動作確認しながら進める
5. 最後に未実装項目・妥協点・注意点を明記する

## 完了条件
以下を満たしたらMVP完了とみなす。
- 4〜8人で一通り最後まで遊べる
- ゲーム進行が破綻しない
- 秘密情報が他人に漏れない
- 同期ずれが致命的に起きない
- 同端末復帰ができる
- 主要ケースのテストまたは確認手順がある
